# 输出契约

## 目录

```text
<实验名>-完成版/
├── T1/
│   ├── T1.dev
│   ├── main.cpp
│   ├── <数据结构>.h
│   ├── <数据结构>.cpp
│   └── <本题测试数据>
├── ...
├── TX/
├── <实验名>-实验报告.docx
├── <实验名>-实验报告.pdf
└── <实验名>-验收问答.pdf
```

每题一个目录且只实现本题。不要提交 `.exe`、`.o`、`.obj`、`.layout`、`.bak`、缓存、渲染 PNG、临时 JSON、个人绝对路径或 Skill 文件。

## 教师代码

优先级：用户同目录教师文件 > DOCX 内可提取附件 > 匹配的内置 starter > 新建自有实现。

- 对教师文件做原样复制。允许在明确的学生区域追加声明和实现，不允许改写学生区域以前的字节、空白、注释、标识符或函数行为。
- 内置 starter 只适用于 `inspect_requirements.py` 明确匹配的五个 profile。不得把不匹配的 starter 强套到新要求。
- 没有教师文件且 profile 不匹配时，建立新的类式实现，并在任务说明中称为“本次实现”，不得伪称“老师所给文件”。

## 代码边界

- `main.cpp`：`#include`、固定测试数据、对象构造、公开方法调用、必要的结果输出。
- `.h`：数据类型、类声明、常量与本题算法入口声明。
- `.cpp`：基础实现和本题算法实现。
- 禁止在 `main.cpp` 中出现类/结构体定义、成员函数定义、图/树/链表/栈/队列核心遍历算法、节点指针重连或排序主体。
- 同一题不携带其他题算法；共用的教师基础运算可以保留。

## 编码

- `.cpp`、`.h/.hpp`、`.c`、`.dev`、`.txt`、`.btr`、`.tre`、`.grp`：严格 GBK（CP936），建议 CRLF。
- DOCX 是 OOXML ZIP 容器，内部 XML 必须遵守 OOXML 的 Unicode 编码；PDF 和 ZIP 是二进制格式，不能声明为 GBK。对这些格式的目标是中文无乱码、文件名可正确解压。
- ZIP 中文文件名使用标准 Unicode 标志；不要用错误的本地代码页伪造 ZIP 条目名。

## 隐私与来源标记

- 不把任何样例学生姓名、学号、班级、电话、地址、文件路径、文档作者或修订者写入 Skill、starter、示例、README 或发布 ZIP。
- 最终实验报告可以保留用户输入文档中原有、提交所必需的信息；不得从其他样例补入身份信息。
- 项目公开联系方式只允许 `15723209543@163.com`。
- 实验交付物中禁止出现 `AI生成`、`由AI生成`、`人工智能生成`、`ChatGPT`、`Codex`、`大模型生成`、`generated by AI` 等来源字样。

## 质量门槛

- 题目数、测试数据和可选/扩展题范围与要求一致。
- 每个 Dev-C++ 工程能打开，工程条目与目录实际文件一致。
- 编译无错误；运行结果来自实际程序。不能运行时明确标记未运行。
- 报告前三部分内容不变；后三部分完整、顺序正确。
- DOCX 每页可视检查通过，实验报告 PDF 来自最终 DOCX 转换且页数合理。
- 验收问答 PDF 与 `T1...TX` 一一对应，不展示源码；每题至少 8 个基于实际实现的短问答，共性问题至少 8 个，运行与调试问题至少 6 个。
- 验收问答覆盖题目、结构、封装、运行链路、复杂度、边界、测试、错误与替代方案，且所有页面可视检查通过。
