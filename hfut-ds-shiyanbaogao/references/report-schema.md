# `report-spec.json` 规范

## 示例结构

```json
{
  "experiment_title": "数据结构实验",
  "shared_types": {
    "title": "本实验设计的共用数据类型",
    "screenshot_placeholder": "【图片位置：插入共用数据类型定义的完整截图，需包含类名、成员变量和必要注释。】",
    "description": ["说明所用存储结构、关键成员、封装关系和共用基础操作。"]
  },
  "problems": [
    {
      "name": "题目1",
      "algorithm_idea": "分析题目约束并说明处理方法。时间复杂度为 O(n)，空间复杂度为 O(1)。",
      "steps": ["检查输入与结构状态。", "调用类中封装的算法并得到结果。"],
      "code_screenshot_placeholder": "【图片位置：插入 T1 对应 .cpp 中本题算法实现的完整截图。】",
      "test_data": ["按实验要求填写实际测试数据。"],
      "result_screenshot_placeholder": "【图片位置：插入 T1 程序运行窗口截图，需同时显示输入、输出和题号。】",
      "result_explanation": ["根据实际运行输出逐项说明结果及边界情况。"]
    }
  ],
  "reflection": {
    "summary": ["总结本次实验完成内容、采用结构、各题结果和总体结论。"],
    "experience": ["结合真实调试过程写心得、困难、错误定位和理解变化。"],
    "suggestion": "提出一条具体、克制且可执行的建议。"
  }
}
```

## 强制规则

- `problems` 数量必须与 `T1...TX` 一致，顺序一一对应。
- `algorithm_idea` 必须同时出现“时间复杂度”和“空间复杂度”，去除空白后不超过 150 个字符。直接给出复杂度结论，不展开复杂度推导。
- `steps` 为 1—10 项。脚本自动添加 `①`—`⑩`；JSON 内不要自己添加序号。每步不得含换行，建议不超过 45 个字符，保证五号宋体下尽量一行显示。
- 共用类型先给截图占位，再解释。每题先算法思想，再算法步骤，最后实现代码截图占位。
- 运行结果每题独立成块，必须含真实测试数据、结果截图占位和结果说明。多组数据全部列出。
- `reflection.summary + reflection.experience + reflection.suggestion` 去除空白后必须超过 800 个字符；`suggestion` 不超过 100 个字符。
- 心得必须基于实际完成过程，例如指针错误、边界用例、递归出口、队列状态、图数据格式或调试方法；不得编造没有发生的经历。
- 禁止来源标记、隐私样例、内部工具名、路径和占位符 `TODO/TBD/待补充`。

## 生成版式

脚本保留原有三个语义章节标题，只插入章节内部内容。新增段落统一：

- 中文字体：宋体；西文字体也设为宋体以避免样式漂移。
- 字号：五号（10.5 pt，即 OOXML `w:sz=21`）。
- 首行缩进：2 字符（`w:firstLineChars=200`）。
- 行距：单倍（`w:spacing w:line=240 w:lineRule=auto`）。
- 段前/段后：0。
- 板块：`（1）【共用数据类型】`、`（2）【题目1】`。
- 内部小标题：`【算法思想】`、`【算法步骤】`、`【实现代码】`、`【测试数据】`、`【结果说明】`。
