# HFUT Data Structures Lab Builder / 数据结构实验代码、报告与验收问答

一个同时适用于 **Codex** 和 **网页版 GPT** 的数据结构实验完整任务包。用户提供实验要求 DOCX 后，可按照题目要求生成分题 Dev-C++ 工程、GBK C++ 代码、测试数据、续写后的实验报告 DOCX、实验报告 PDF、面对面验收问答 PDF，以及最终完成版 ZIP。

本 README 同时适用于以下两个发行包：

- `HFUT数据结构实验报告-Codex版-v2.2.zip`
- `HFUT数据结构实验报告-网页版GPT-v2.2.zip`

两个版本使用同一套核心工作流，区别主要在于调用入口：Codex 版通过安装 Skill 后调用，网页版 GPT 版通过上传压缩包和实验要求直接执行。

## 2. Skill 说明

作为学长，我想先说明，这个 Skill 的主要作用，是帮助大家更快地完成数据结构实验中的重复性工作，例如整理实验要求、建立分题工程、生成基础代码、组织测试数据、续写实验报告以及准备验收问答。它能够提高实验完成效率，减少文件命名、工程配置、编码格式、报告排版等方面的机械工作，但它对数据结构能力本身的提升非常有限。Skill 可以帮助你把一份实验“做出来”，却不能保证你真正理解顺序表为什么需要移动元素、单链表删除结点时指针如何变化、栈和队列的操作顺序、树的递归过程以及图的搜索原理。生成的代码能够运行，也不等于这些代码已经变成了你自己的知识。因此，这个项目只能作为实验助手使用，不能代替听课、看书、调试代码和独立思考。拿到生成结果后，至少应当把每一道题亲自运行一次，了解类中保存了哪些数据、调用了哪些成员函数、程序按照什么顺序执行，以及测试结果为什么正确。

其次，必须重视老师的面对面验收。不同老师的验收方式差别很大，有些老师只会简单询问实验功能（如ljw），有些老师则会进行连续、快速、层层深入的追问（如zb）。老师可能刚问完“这道题用了什么数据结构”，马上继续追问“为什么不用另一种结构”“这个函数的时间复杂度是多少”“循环什么时候结束”“指针修改顺序能不能交换”“如果输入为空会发生什么”“这组测试数据验证了哪种边界情况”，只要你一旦没有回答出来或者是回答错误就让你报学号了。这种验收方式就像连续输出问题，如果平时只是复制代码、照着报告背答案，很容易在第二个或第三个追问时回答不上来。验收问答 PDF 只能帮助大家整理常见问题，不能保证老师一定按照其中的问题提问。真正有效的准备方式，是对照实际代码理解每个成员变量、关键循环、递归出口、指针变化和测试用例，并能够用自己的话解释。尤其是顺序表、链表、栈、队列、树和图这些基础内容，老师很容易从一个简单问题不断向底层实现追问。

不过，验收时也没有必要过度紧张。根据以往情况，大多数老师并不是故意为难学生，而是想确认实验是不是本人完成、学生是否掌握了最基本的实现思路。即使遇到老师连续提问，也要先稳住心态，听清问题后再回答，不要因为第一个问题没听懂就慌乱。回答时可以先说结论，再请求老师能否结合代码说明原因（不是每一个老师都允许让你看着代码验收）；暂时想不起来时，也可以先描述自己的实现流程，而不是直接沉默。老师通常会根据代码完成情况、回答情况和报告质量综合评分，只要实验确实完成、基本思路能够说清楚、态度认真（这点非常关键，一定要非常诚恳，不要不懂装懂），最终得分一般不会太低（有人上去说啥也不会都有81分）。验收前最好为每道题准备一句话目标、三到五句算法流程，以及一个能够说明正确性的测试用例，这样即使老师改变提问方式，也能从自己理解的内容出发回答。

最后，实验报告的格式一定要规范。数据结构实验不仅检查程序能否运行，也会检查报告是否按照模板填写。标题层级、字体字号、段落缩进、行距、算法步骤、复杂度分析、代码位置、运行截图和总结内容都应符合教师要求。不能随意删除模板原有内容，也不能把截图占位文字直接保留在最终报告中。运行结果截图必须来自自己实际运行的程序，图片要清晰，能够看到输入、输出和关键结果；每张截图后还应配有必要说明，不能只放图片不解释。代码、截图和报告描述必须相互对应，不能出现报告写的是一种算法，而工程中实现的是另一种算法的情况。提交前应重新检查文件名、题号、工程数量、代码编码、报告页码、图片是否完整以及压缩包能否正常解压。特别说明一下，如果排版不好看，报告内容写得再好一定是低分。

总的来说，这个 Skill 能够帮助大家快速搭建实验成果、减少重复劳动，但不能代替数据结构学习。使用时应当明确顺序：先借助工具完成基础内容，再亲自编译运行，接着理解代码和准备验收，最后检查报告格式并提交。不要把它当成一键完成课程任务的工具，而应把它作为整理实验和辅助复习的手段。真正决定实验验收和课程成绩的，仍然是你是否能够看懂代码、解释算法、应对追问，并提交一份内容完整、格式规范的实验报告。

最后，我想说：数据结构作为一门计算机专业最基本的专业课，希望同学们还是要认真学习，如果仅仅是使用ai完成实验任务肯定是达不到学习效果的，请谨慎使用本skill。

## 3. 功能概述

本项目从实验要求出发，把题目、代码、测试、报告、验收复习和最终打包组织成同一套可校验流程。

主要功能包括：

- 读取并解析实验要求 DOCX；
- 提取实验标题、实验任务、题目数量、测试数据和报告章节；
- 根据题目数量建立 `T1...TX` 独立目录；
- 为每道题生成 Dev-C++ 工程；
- 使用类和对象封装数据结构与算法；
- 保持 `main.cpp` 只负责测试数据准备和成员函数调用；
- 优先使用教师提供的 `.h/.cpp`、测试数据和其他附件；
- 使用哈希校验保护教师代码前缀不被改写；
- 严格生成 GBK / CP936 编码的源码、工程文件和文本测试数据；
- 复制原 DOCX 后，只完成指定报告章节；
- 生成实验报告 PDF，并逐页检查版面；
- 根据实际代码和测试结果生成面对面验收问答 PDF；
- 校验工程数量、代码编码、报告格式、问答题量、隐私信息和压缩包完整性；
- 输出可直接检查和提交的最终完成版 ZIP。

## 4. 面对面验收问答 PDF

完成实验后，任务包会额外生成：

```text
<实验名>-验收问答.pdf
```

这份 PDF 不用于替代源码和实验报告，而是帮助学生在验收前快速整理实验思路。问答内容会根据本次实验的真实代码、类、成员函数、算法流程、循环、递归、指针、动态内存、栈、队列、矩阵和测试结果生成，不是与代码无关的固定题库。

每份验收问答手册通常包括：

- 实验主题、逻辑结构和存储结构；
- 类与对象的封装方式；
- 核心成员变量和成员函数；
- 至少 8 个基础与共性问题；
- 每道题 8—16 个模拟问题；
- 每道题的目标、实现主线和边界情况；
- 时间复杂度与空间复杂度；
- 循环或递归过程；
- 指针、下标、栈顶、队头、队尾等状态变化；
- 至少 6 个运行、测试和调试追问；
- 常见错误、替代方案和验收前检查清单；
- 每道题约 30 秒的口述总结。

单个答案通常控制在 30—100 字，原则上不超过 220 字，采用“先说结论，再解释原因”的表达方式。建议学生先遮住答案独立回答，再回到实际工程中查找对应代码，不能只背 PDF 中的文字。

## 9. 代码规则

### 9.1 工程划分

- 每道题建立一个独立 `TX` 文件夹；
- 每个 `TX` 目录只保留一个 Dev-C++ 工程；
- 工程名与题号保持一致；
- 每道题只实现本题要求的算法；
- 共用的教师基础运算可以保留，但不能把其他题算法混入当前工程。

### 9.2 类与对象

- 数据结构应使用类或对象进行封装；
- `.h` 文件保存类声明、结构体定义和函数声明；
- `.cpp` 文件保存成员函数和算法实现；
- `main.cpp` 只负责准备测试数据、创建对象和调用公开成员函数；
- 不应把核心算法全部堆放在 `main.cpp` 中。

### 9.3 教师代码保护

教师提供的代码必须优先使用。教师代码前缀保持原字节不变，学生新增声明和实现写入指定区域。工作流通过哈希值检查教师代码是否被误改。

### 9.4 注释要求

关键位置应添加简洁中文注释，包括：

- 关键分支；
- 循环不变式；
- 递归出口；
- 指针变化；
- 队头、队尾、栈顶变化；
- 图搜索中的访问标记；
- 复杂边界条件。

注释应解释设计意图，不应把每行代码机械翻译成中文。

### 9.5 编码要求

以下文件严格使用 GBK / CP936：

```text
.cpp
.h
.dev
文本测试数据
```

不能把 UTF-8 源码直接伪装成 GBK，也不能通过忽略编码错误的方式强行写出文件。

## 10. 实验报告规则

### 10.1 原模板保护

- 先复制原 DOCX，再进行填写；
- 禁止覆盖原文件；
- 实验目的、实验要求、实验任务等前三个语义部分保持不变；
- 只填写指定的后三个语义章节。

### 10.2 每道题内容

每道题至少包含：

```text
【算法思想】
【算法步骤】
【实现代码】
```

算法思想应直接说明使用的数据结构、核心算法和实现目标，原则上不超过 150 字，并给出时间复杂度和空间复杂度。

算法步骤使用：

```text
① ② ③ ④ ⑤ ⑥ ⑦ ⑧ ⑨ ⑩
```

每题最多十步，避免把源码逐行改写成步骤。

### 10.3 代码与截图

共用数据类型、实现代码和运行结果应放置明确的图片或文字占位。最终提交前，学生需要将运行结果占位替换为自己实际运行得到的截图。

截图要求：

- 清晰可读；
- 能看到输入和输出；
- 能证明本题功能正确；
- 与当前题号和当前代码一致；
- 截图后有必要的结果说明。

### 10.4 总结要求

“总结、心得和建议”合计应超过 800 字，其中建议部分一般不超过 100 字。内容应结合本次实验真实过程，说明：

- 学到了什么；
- 遇到了什么问题；
- 如何调试和解决；
- 对数据结构理解有什么变化；
- 对实验安排有什么合理建议。

### 10.5 排版要求

新增文字统一使用：

- 宋体；
- 五号；
- 首行缩进 2 字符；
- 单倍行距。

若教师模板另有明确要求，以教师模板为准。

## 13. 编码、依赖与使用边界

### 13.1 编码规则

- `.cpp/.h/.dev` 与文本测试数据：GBK / CP936；
- README、Python 脚本与配置文件：UTF-8；
- DOCX、PDF 与 ZIP：按各自标准格式保存；
- 中文文件名和中文正文不得出现乱码。

### 13.2 PDF 转换

实验报告 PDF 转换优先使用：

1. Microsoft Word；
2. LibreOffice。

验收问答 PDF 使用 ReportLab 生成，并通过 Poppler 或 PyMuPDF 渲染检查。

### 13.3 编译与验证边界

如果当前环境没有可用的 C++ 编译器，只能完成静态检查，并必须明确标记：

```text
未编译
```

## 18. 结语

HFUT Data Structures Lab Builder 的目标，是把实验要求、代码工程、测试数据、实验报告、验收复习和最终打包组织成一套前后对应的流程，帮助学生减少重复劳动和低级格式错误。

但无论工具多么完整，它都不能代替真正的数据结构学习。只有亲自运行程序、理解每一种结构的存储方式、弄清每一步操作造成的状态变化、能够解释算法复杂度和边界条件，才能在验收和考试中独立回答问题。

建议始终按照以下顺序使用本项目：

```text
读取要求 → 生成基础成果 → 亲自编译运行 → 理解代码 → 补充真实截图 → 准备验收 → 检查格式 → 最终提交
```

希望大家把这个 Skill 当作实验助手和复习框架，而不是一键提交工具。工具帮助你节省整理时间，真正决定实验质量和课程收获的，仍然是你对代码和数据结构的理解。
